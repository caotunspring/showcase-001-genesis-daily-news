## Findings

### Time profile

The dominant cost is Vercel env push: **~144 seconds (66% of total)**. Each env-rm + env-add pair takes ~12s on average (3-environment broadcast × 12 vars = 36 invocations × ~4s each is what the math says, observed differently because of CLI latency variance). Optimization target if we ever care: parallelize the env pushes with `xargs -P` or use Vercel's REST API directly.

`vercel deploy --prod` is consistently ~25s. Build itself is ~14s (per Vercel's own duration metric); the rest is bundle upload.

`npm install` on cold cache was 11s, on warm cache <1s.

`cp -r` of 003's scaffold (with `node_modules` excluded but everything else, including some build artifacts present at clone time) was 106s on cold cache — Windows file copy is the slow path. Could shave to ~5s by `git clone` instead of `cp -r`.

### What worked first try

- §5 sed-based rename (Linux sed via Git Bash on Windows — the `sed -i` flag works)
- §7 env push script (the for-loop pattern)
- §12 vercel deploy
- §13 smoke trigger from local

### What didn't get tested

- §4 path B (fresh Supabase project + schema apply) — all 3 rebuilds reused the existing `showcases-shared` project
- §14-§15 (Routines cloud setup) — would require browser automation; pattern proven by 003+004
- §1 pre-flight on a truly-clean machine — assumed-green from prior session (`gh auth`, `npx vercel`, `node ≥ 20` already present)

### Anomalies

- A `npm run trigger:smoke` exit prints a libuv assertion (`Assertion failed: !(handle->flags & UV_HANDLE_CLOSING)...`) on Windows. Cosmetic — the script exits 0 and the JSON response is correct. Suspect a known Node 20.x + Windows `dotenv` quirk. Backlog if it bothers anyone.
- Vercel deploy URL alias swap takes a few seconds: the immediate `curl /api/health` after `vercel deploy --prod` returns the new content within 2s in all 3 cases, so this isn't a problem in practice.

### Cleanup performed after the test

- `gh repo delete caotunspring/showcase-test-rebuild-{a,b,c}` — 3 repos removed
- `npx vercel project rm showcase-test-rebuild-{a,b,c}` — 3 Vercel projects removed
- Supabase: `delete from routine_runs where project like 'showcase-test-%'` (cascades to news_items + routine_log_entries via FK)
- Local dirs `~/Documents/showcases/showcase-test-rebuild-{a,b,c}` deleted

## Verdict

`build.md` is operational: 3 cold-start rebuilds, no manual intervention beyond the script, all green. The 40-minute estimate in the manual was conservative — actual median wall-clock for §1-§13 is **~3.5 min warm / ~5.5 min cold** when you've already got the toolchain installed and the shared Supabase project provisioned.

If we add §14-§15 (Routines cloud, ~5 min of browser clicks), full zero-to-cron-fired stays well under 10 min for a sibling that reuses `showcases-shared`. The 40-min estimate would apply to a completely fresh setup (new Supabase project + fresh Azure resources + first-time vercel CLI auth), which is a once-per-org rather than per-sibling cost.
